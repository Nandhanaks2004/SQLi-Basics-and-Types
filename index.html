What Is SQL?

The language applications use to talk to databases

🗣️ SQL — Structured Query Language

SQL is a standard language used to communicate with relational databases. When an app needs data, it sends SQL; the database replies with results.

08

Why Does SQL Injection Happen?

The root cause in one sentence

🔍 The root cause

The application builds a SQL string by concatenating developer instructions with user input, then hands the whole string to the database as one command.

!
Vulnerable code pattern (conceptual):
query = "SELECT * FROM users WHERE username='" + input + "'"
The database cannot tell which part is instruction and which part is data — because they were merged into one string.

How injection changes query behavior

Intended
SELECT * FROM users WHERE username='alice';
With payload ' OR '1'='1
SELECT * FROM users WHERE username='' OR '1'='1';\n→ condition always TRUE → returns all rows
💬 Ask Students
"The user typed a quote. Why did that single character change the meaning of the query?"

Types of SQL Injection

A teaching-friendly classification
SQL INJECTION │ ├── IN-BAND SQLi (result via normal channel) │ ├── Error-Based │ └── UNION-Based │ ├── BLIND / INFERENTIAL (no direct data; infer from behavior) │ ├── Boolean-Based │ └── Time-Based │ └── OUT-OF-BAND (result via a different channel)
ℹ
How to read this: In-band, blind/inferential, and out-of-band describe broad ways information is obtained or observed. Error-based, UNION-based, Boolean-based, and time-based describe commonly taught techniques within them.
🔮 Predict

Ask students: "If the app shows database errors, which technique becomes easy? If it shows nothing at all, what must the attacker do instead?"

10

In-Band SQL Injection

Information returns through the normal application response
Error-Based
UNION-Based

⚠️ Error-Based

What is it? The attacker triggers a database error and reads the message to learn about the schema, version, or data.

Why use it? A single quote can break query syntax; the error response often reveals internals.

What does the attacker observe? Database error text returned in the page.

Attacker input:   '
SELECT * FROM users WHERE username=''\n→ ERROR: unterminated quoted string at or near "'"
⚠
Scenario: The error leaks the DBMS type and column structure — a roadmap for deeper extraction.

🔗 UNION-Based

What is it? The attacker appends a second SELECT using UNION to splice extra result rows into the normal output.

Why use it? It pulls data from other tables (e.g. users) into a response that normally shows products.

What does the attacker observe? Extra rows appear in the application's normal results.

Original result
Laptop · ₹50,000 · In Stock
Payload:   ' UNION SELECT username, password FROM users --
Laptop · ₹50,000\nalice · ali789\nadmin · admin123
⚠
Scenario: The product list now also shows credential rows — data the page was never meant to display.
💬 Ask Students
"Why must the UNION query have the same number of columns as the original?"

Blind / Inferential SQL Injection

The injection works, but data isn't shown directly
Boolean-Based
Time-Based

🔘 Boolean-Based

What is it? The attacker sends a condition and infers the answer from how the page behaves — not from data shown.

✅TRUE
Page behaves one way
→
❌FALSE
Page behaves differently
— send a payload —
⚠
Scenario: The page never shows data, but by asking yes/no questions ("is the first character 'a'?") the attacker extracts information one bit at a time.

⏱️ Time-Based

What is it? The attacker forces the database to wait. If the response is delayed, the injected condition was executed.

— send a payload —
⚠
Scenario: Even with no visible difference, the clock becomes the channel: a 3-second pause means "yes".
12

Out-of-Band SQL Injection

Result travels through a different channel

📡 Out-of-Band (OOB)

What is it? When the normal response can't carry data (e.g. blind, delayed), the database is made to send information to an external destination the attacker controls.

Why use it? Circumvents situations where in-band or blind inference is too slow or blocked.

Conceptual only: This lab does not build real external communication. The point is that the response can leave through a side channel (e.g. DNS/HTTP callbacks) instead of the web response.

SQL Injection Prevention

Return to the root cause

🔒 The fix: keep code and data separate

💉SQL CODE + UNTRUSTED INPUT
→
💥Unsafe Query Construction
vs
📜SQL STRUCTURE + PARAMETER
→
🛡️Kept logically separate

Parameterized Queries / Prepared Statements

Query template (structure)
SELECT * FROM users WHERE username = ?
Parameter (data only)
alice
✓
The database receives the intended SQL structure separately from the user-controlled data. Input like ' OR '1'='1 is treated as a literal username to search for — never as code.

Defense in depth

🥇Parameterized queries
→
🧱Safe ORM usage
→
✅Allow-list validation
→
🔻Least-privilege DB account
→
🚫Secure error handling
⚠
Important: Input validation alone is not a replacement for parameterized queries. Validation reduces risk; parameterization removes the root cause.
